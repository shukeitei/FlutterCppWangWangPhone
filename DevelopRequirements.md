# 汪汪机开发需求文档

## 用户旅程划分

### 阶段一：启动与解锁

#### 1.1 开屏动画
- 应用启动时的品牌动画展示
- 动画结束后自动过渡到锁屏界面
- 动画时长：约2-3秒

#### 1.2 锁屏界面
- 显示时间、日期
- 显示通知预览（如有新消息）
- 用户点击屏幕后显示密码输入界面
- **UI设计**：使用磨砂半透明玻璃效果提升质感

#### 1.3 密码解锁
- 六位数字密码输入
- 密码验证通过后进入主屏幕
- 支持密码修改功能
- 首次使用时引导用户设置密码
- **UI设计**：使用毛玻璃效果，输入框采用磨砂质感

---

### 阶段二：主屏幕

#### 2.1 桌面主屏幕
- 显示应用图标网格
- 底部应用栏（类似手机 Dock 栏）
- 支持滑动切换多页桌面
- 显示天气组件（可选）
- **UI设计**：使用毛玻璃效果背景

#### 2.1.1 天气小组件（可选）
- 显示在桌面上的小型天气信息
- 显示当前温度和天气图标
- 点击可展开查看详情或打开天气应用

#### 2.3 应用图标交互
- **点击**：打开对应应用
- **长按**：进入编辑模式，可删除或移动图标
- **初始应用**：仅在主屏幕放置三个应用图标
  - 聊天App（微信）
  - 设置App
  - 天气App
- **设计目的**：减少开发压力，聚焦核心功能

#### 2.1.2 天气应用
- **应用类型**：独立应用（类似手机系统天气App）
- **数据来源**：7timer免费天气API
- **API地址**：http://www.7timer.info/bin/api.php
- **数据格式**：JSON（固定）
- **产品类型**：civillight（一周逐日预报）
- **功能特点**：
  - 显示当前天气状况（天气类型图标）
  - 显示当前温度、城市名称
  - 显示湿度、风向、风速等详细信息
  - 一周7天天气预报
  - 支持手动设置经纬度位置
  - 支持温度单位切换（摄氏度/华氏度）
- **定位功能**：
  - 自动获取用户GPS定位（精确经纬度）
  - IP定位获取用户大致位置（城市级别）
  - 经纬度逆地理编码获取城市名称
  - 支持定位权限管理
  - 定位失败时使用默认位置

#### 2.2 状态栏
- **设计说明**：直接使用用户设备上的系统默认状态栏
- 不在应用内重复创建虚拟状态栏
- 应用内容从状态栏下方开始布局
- 确保内容不被状态栏遮挡（使用 SafeArea）

### 阶段三：应用功能

#### 3.1 微信应用

##### 3.1.1 底部导航栏
- **聊天**：消息列表和聊天界面
- **联系人**：AI角色列表和管理
- **朋友圈**：动态发布和浏览
- **我**：个人资料和设置

##### 3.1.2 聊天模块
- **消息列表展示**：
  - 以微信会话页为参考，展示最近聊天的联系人/群聊列表。
  - 每个会话项需显示头像、名称、最后一条消息摘要、最后活跃时间、未读数量。
  - 消息摘要需根据内容类型动态显示，例如“文字内容”“[语音]”“[图片]”。
  - 支持按照最近消息时间倒序排列，保证活跃会话优先展示。
  - 当AI主动发来消息时，列表需自动刷新对应会话顺序，并在锁屏/桌面通知区域展示预览。
  - 支持单聊与群聊两种会话类型，群聊需要额外显示群名称与群成员发言者。
- **聊天详情界面**：
  - 顶部导航栏显示联系人名称/群名称、头像、返回按钮，可预留“更多”入口。
  - 中间区域为消息气泡流，按时间顺序展示历史消息，区分用户消息与AI消息的左右布局。
  - 不同消息类型使用不同气泡样式：文字消息、语音消息、图片消息要有明确视觉区分。
  - 长对话场景下需支持懒加载历史记录，避免一次性加载过多消息导致卡顿。
  - 需支持展示消息发送状态，例如发送中、发送成功、生成失败，方便用户理解AI生成过程。
  - 群聊中每条AI消息需显示发言角色昵称，增强多角色社交网络的真实感。
  - 聊天气泡不能只按“用户/AI”二分，还要按 LLM 输出的组件类型进行渲染，确保结构化消息能以接近真实IM的方式展示。
- **消息解析与前端渲染机制**：
  - LLM 输出严格遵循 [`asset/roleplay_prompt_v4_optimized.txt`](asset/roleplay_prompt_v4_optimized.txt) 中定义的 JSON Schema，前端与C++核心层都需要基于 `type` 字段实现统一解析器。
  - 解析器需支持至少以下消息类型：`thought`、`word`、`action`、`state`、`accept`、`accept_redpacket`、`accept_transfer`、`reject`、`reject_redpacket`、`reject_transfer`、`memory`、`redpacket`、`transfer`、`emoji`、`location`、`image`、`note`、`product`、`link`、`diary`、`moment`、`moment_comment`、`moment_like`、`summary`、`system`。
  - 解析结果不能直接原样平铺显示，必须先映射为应用内部统一消息模型，再交给 Flutter 聊天气泡组件树进行差异化渲染。
  - 需要区分“用户可见组件”和“系统内部组件”：如 `word`、`action`、`emoji`、`image`、`redpacket`、`transfer` 等属于可见消息；`thought`、`summary`、`memory`、`diary`、`system` 更多用于状态驱动、上下文管理或特殊控制，不应直接按普通聊天文本展示。
  - 当解析器遇到未知 `type`、字段缺失、JSON格式错误时，需进入容错逻辑：记录日志、保留原始响应、避免聊天页面崩溃，并向上层返回可调试状态。
  - 解析器设计需由 C++ 核心层负责协议解析和标准化数据输出，Flutter 层负责根据标准化后的消息类型渲染浅色/深色模式一致的聊天气泡。
- **不同类型消息的前端展示要求**：
  - `word`：显示为标准文字气泡，支持多条连续短消息拆分展示，支持 `ref` 引用样式。
  - `action`：显示为弱化样式的动作气泡或舞台描述气泡，与普通文本视觉区分，避免误认为口语内容。
  - `emoji`：显示为独立表情气泡或大表情内容块，并根据表情ID映射到本地表情资源。
  - `image`：显示为图片卡片气泡，支持缩略图、点击查看大图、加载中、失败态占位。
  - `redpacket` 与 `transfer`：显示为红包/转账专用卡片气泡，并支持状态流转，如待领取、已领取、已拒收。
  - `accept*` 与 `reject*`：属于资金类状态消息，需要驱动红包/转账卡片状态更新，而不是单独显示成普通文字。
  - `location`：显示为位置卡片，包含地点名称，可预留地图详情入口。
  - `note`：显示为备忘录样式卡片，可用于提醒、约定、待办等场景。
  - `product`：显示为商品分享卡片，包含商品名、价格、图片。
  - `link`：显示为链接预览卡片，包含标题和链接地址。
  - `moment`、`moment_comment`、`moment_like`：原则上不直接作为普通聊天气泡展示，而应驱动朋友圈模块的数据变化；聊天页可按需要展示“朋友圈互动提示消息”。
  - `state`：用于更新AI当前情绪状态，可影响头像挂件、输入中状态、后续气泡风格或调试面板显示，不建议直接作为文本消息展示。
  - `thought`、`summary`、`memory`、`diary`：默认不直接展示在聊天气泡中，但必须存储并进入上下文系统，供后续推理、调试和问题排查使用。
  - `system`：当内容为 `empty` 时表示本轮沉默，不产生普通聊天气泡，但前端要正确处理该轮结束状态，避免用户误认为系统卡死。
- **发送文字消息**：
  - 底部输入区提供文本输入框、发送按钮，交互逻辑尽量贴近真实聊天App。
  - 用户发送后，消息先以“本地已发送”状态插入列表，再调用聊天API生成AI回复。
  - 支持回车发送或点击按钮发送，空文本不可发送。
  - 文字消息需写入本地SQLite数据库，保证重启应用后聊天记录可恢复。
  - AI回复应支持流式返回，逐步展示内容，提升真实陪伴感。
  - 后续可扩展表情、引用回复、撤回等能力，但当前版本先聚焦基础文字对话闭环。
- **发送语音消息（需配置语音API）**：
  - 用户侧可先支持“点击发送预设语音消息”或“长按录音”两种方案，最终实现以产品开发进度为准。
  - AI侧语音消息不直接存储真实录音，而是基于语音API生成音频文件或音频链接，再以语音气泡形式展示。
  - 若用户未在设置中配置语音API，则聊天界面中语音发送入口置灰，点击后提示用户先完成配置。
  - 语音消息需显示时长、播放状态，并支持点击播放/暂停。
  - 音频文件路径、时长、生成状态等信息需纳入本地数据库字段，便于恢复与调试。
  - 语音生成失败时，需在消息气泡中明确提示失败原因，并支持重新生成。
- **发送图片（需配置生图API，通过设置中的生图API实现）**：
  - 图片消息主要用于AI根据聊天上下文主动发送“自拍照”“场景图”“分享图片”等内容，增强沉浸感。
  - 生成流程为：聊天上下文整理 → 生成图片提示词 → 调用设置页配置的生图API → 返回图片地址/本地缓存路径 → 插入聊天记录。
  - 若未配置生图API，则图片发送能力默认关闭，并在相关入口提示用户前往设置页完成配置。
  - 图片消息在聊天界面中以缩略图卡片展示，点击后可查看大图。
  - 需支持图片加载中、生成中、生成失败三种状态，避免用户误以为功能无响应。
  - 图片相关数据需持久化保存，包括提示词、图片地址、本地缓存地址、生成时间，便于后续复用与排查问题。

##### 3.1.3 聊天上下文管理机制
- **目标说明**：每次用户发送新消息时，系统都必须重新组装一次完整推理上下文，确保角色一致性、世界观一致性、上下文连贯性，以及组件输出协议稳定。
- **上下文分层**：
  - System Prompt：负责提供规则、设定、长期记忆、动态状态。
  - User Prompt：负责提供当前这一轮需要回复的聊天现场。
- **System Prompt 内容顺序**：
  1. 当前系统日期
  2. 当前系统时间
  3. 当前系统天气
  4. 未来天气预报
  5. 主系统提示词 （roleplay_prompt_v4_optimized.txt）
  6. AI角色人设
  7. 用户人设
  8. 世界书
  9. 预设
  10. 动态summary
  11. AI角色记忆memory
  12. 可用表情包列表
- **User Prompt 内容**：
  - 动态截取的聊天记录。
- **动态summary规则**：
  - summary 必须跟随每一轮AI回复实时更新，这一点与 [`summary`](asset/roleplay_prompt_v4_optimized.txt:157) 的定义保持一致。
  - summary 采用自然语言文本直接存储，不额外拆分为结构化字段。
  - summary 的职责是承接更早的对话内容、当前话题、双方情绪、未完成事项和关键事件，用于下一轮请求时恢复上下文。
- **memory规则**：
  - memory 属于长期记忆，用于保存用户偏好、关系信息、约定、纪念日、重要事项等长期有效内容。
  - memory 与 summary 需要同时注入 system prompt，其中 summary 负责动态上下文，memory 负责长期记忆。
- **聊天记录截取规则**：
  - 聊天记录的截取条数由用户在设置中配置。
  - 每次请求仅将设置范围内的最近聊天记录注入 User Prompt。
  - 更早的聊天内容不再直接拼入 User Prompt，而是由动态summary承接，形成完整上下文闭环。
- **世界书与预设规则**：
  - 世界书与预设均为纯文本内容。
  - 不采用按关键词命中注入，也不采用酒馆类的分段触发机制。
  - 每次请求时，世界书与预设都作为固定 system prompt 内容参与拼接。
- **可用表情包规则**：
  - 表情包列表需提供表情ID以及对应的语义说明、情绪表达或使用场景说明。
  - 该列表作为 system prompt 的组成部分，用于约束和引导模型输出正确的表情包组件。
- **设计价值**：
  - 将 system 规则层与聊天现场层分离，方便C++核心层统一进行上下文组装。
  - 通过“最近聊天记录 + 动态summary + memory”组合，在控制上下文长度的同时保持长期陪伴感。
  - 为后续 Flutter 聊天界面渲染、C++层上下文拼装、SQLite存储、API请求体生成提供统一标准。
